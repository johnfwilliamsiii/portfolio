01  /  Executive Summary
The opportunity
in plain language
Lakeshore Financial Group currently manages more than $280 million in annualized production across four lines of business using disconnected Excel spreadsheets, a manual weekly consolidation process, and no shared data infrastructure. This proposal describes a purpose-built operations and reporting platform that eliminates that overhead — hosted entirely within our existing Microsoft 365 subscription, at no new licensing cost.
The core argument

We already pay for the infrastructure. Microsoft 365 Business includes SharePoint Online, Power Automate, and the full suite of tools required to build this system. The only incremental cost is implementation. Once built, this platform centralizes all submission data, automates NIGO tracking and alerts, delivers a real-time executive dashboard, and gives each of our 106 agents their own window into their book of business — replacing four separate Excel sheets and a labor-intensive weekly merge with a single source of truth.

What we are proposing

A five-phase build of a unified operations platform hosted on SharePoint Online. The system comprises three smart data entry forms (Fixed Annuity, Broker-Dealer/Independent Advisor, and Life Insurance), a centralized Microsoft Lists database, a live executive production dashboard, and a dedicated Agent Dashboard that gives every producing agent real-time visibility into their own submissions, NIGO items, client roster, and YTD production.

Phase 1 development work is already underway. A fully functional Fixed Annuity entry form prototype has been built and is available for review. The remaining phases follow a structured rollout requiring a Microsoft-certified SharePoint contractor for integration work estimated at 8–12 weeks total.

04  /  Strategic Benefits
Four pillars driving
the business case
The operational improvements are significant. But the deeper strategic value lies in what becomes possible once data is centralized, current, and accessible.
Compliance & audit readiness

Regulatory exposure from suitability data inconsistency is eliminated. Every client's KYC profile is stored once, linked to all their transactions, and timestamped. NIGO items are tracked automatically with age alerts at 30 and 60 days.

  • Suitability data stored once per client — not per transaction
  • Automated NIGO alerts before items reach compliance thresholds
  • Full audit trail on every submission, edit, and status change
  • Role-based access logs — who saw what, when
Real-time data & reporting flexibility

Management sees production as it happens, not as it was four days ago. Any report that can be defined can be run instantly — by agent, by LOB, by week, by state, by carrier, or any combination.

  • Live production totals between Monday and Thursday
  • Instant LOB breakdowns: Fixed, BD, IA, Life
  • Geographic production map — MI, TX, CA, and beyond
  • Historical trend queries going back to day one
Processing speed & staff leverage

The current weekly consolidation process — estimated at 3–4 hours of ops staff time — disappears entirely. The forms guide users through complex logic (flow flip, agent splits, transfer type) with built-in validation, reducing entry errors at source.

  • Manual weekly merge eliminated — ~200 hours/year recovered
  • Conditional form logic prevents common entry errors
  • NIGO email alerts reduce status-check call volume
  • All four ops staff enter to one shared system simultaneously
Agent transparency & retention

Giving agents visibility into their own book of business is a meaningful differentiator in producer recruitment and retention. Agents who can see their pipeline, their NIGO items, and their YTD production in real time are better positioned to serve clients and grow their practice.

  • No more calling ops for a status update
  • Agents see their NIGO items before they become problems
  • YTD totals and pacing visible at any time
  • Competitive advantage in producer recruitment
05  /  The Agent Dashboard
Real-time transparency
for every producing agent
The Agent Dashboard is a dedicated SharePoint page — separate from the operations site — accessible to all 106 agents via their existing Microsoft 365 logins. Each agent sees only their own data, filtered automatically by their identity.
Why this matters

Today, every status inquiry from an agent — "Did my transfer go through?" "Why is my client's application still pending?" "What's my YTD number?" — requires a phone call or email to operations staff. With the Agent Dashboard, those questions are answered the moment an agent opens a browser. For a 106-agent firm, eliminating even two status calls per agent per week reclaims over 10,000 minutes of operations capacity annually.

What agents see

Submitted transactions — all statuses
Every submission the agent is credited on, filterable by date range, transaction type, LOB, and status. Current week, YTD, or full history.
NIGO items — what's stuck and why
A live view of every open NIGO in the agent's book, sorted by age. Notes from operations are visible, giving agents the context they need to follow up with clients directly.
YTD production totals
Credited production broken down by LOB — Fixed, BD, IA, Life — with periodic and transfer subtotals. Updated in real time as records are entered by operations.
Client roster
Every client the agent has written business for, with transaction count and most recent activity date. Foundation for future relationship management features.

What agents can do

The Agent Dashboard is primarily a read-only view. Agents can add notes to their own records — useful for flagging follow-up actions or adding context for operations staff — but cannot edit production figures, statuses, or other agents' records. This keeps data integrity with operations while giving agents a meaningful channel to communicate.

Access model

The Agent Dashboard lives on a separate SharePoint site from the Operations site. Agents log in with their existing Microsoft 365 credentials — no new accounts, no passwords to manage. SharePoint's native audience targeting ensures each agent's view is automatically filtered to their own book of business. No agent can see another agent's data.

Design decisions for the implementing developer

  • Suitability data lives on the Client record, not the transaction. Income, net worth, and investment experience are per-person attributes. Storing them in the Clients list eliminates the current problem of the same client's profile being entered — and potentially entered inconsistently — on every BD/IA transaction.
  • BD and IA share one list. A single LOB flag column (values: BD or IA) distinguishes transaction type. IA-specific fields (custodian, model, advisory fee %) are nullable for BD rows. This is cleaner than two nearly-identical lists and simplifies cross-LOB reporting.
  • NIGO is a filtered view, not a separate list. The NIGO tracker is a saved SharePoint view on the Fixed transactions list, filtered to NIGO_Status = 'NIGO'. No data duplication. The dashboard and Agent Dashboard read the same filtered endpoint.
  • Agent splits stored as flat columns. Agent1Name, Agent1PctSplit, Agent1TotalAmt … through Agent5 (Fixed) or Agent4 (BD/IA/Life). SharePoint Lists does not support row-level child records natively without additional complexity. Flat columns are sufficient for the reporting requirements and simplify form-to-list writes.
  • Week tagging is automatic. Power Automate calculates and stamps WeekNumber and SubmitDate on every record at the time of creation. Ops staff do not enter this manually.

Power Automate flows required

  • Form submission handler (×3 — one per LOB): Receives HTTP POST from the form, validates required fields, looks up or creates the Client record, writes the transaction row to the appropriate list, stamps WeekNumber.
  • NIGO alert flow: Scheduled daily. Queries Fixed transactions where Status = NIGO and calculates days since IGO date. Sends email to the assigned ops processor and CC: operations manager for items exceeding 30 days; escalation email for items exceeding 60 days.
  • Agent dashboard data feed: Triggered on list item create/modify. Updates a pre-aggregated Agent Summary list that powers the dashboard's YTD totals without querying full transaction history on every page load.

API connectivity — dashboard to SharePoint

The executive dashboard and agent dashboard read data from SharePoint Lists using the SharePoint REST API. The standard endpoint pattern is:

GET https://[tenant].sharepoint.com/sites/LakeshoreOperations/_api/web/lists
  /getbytitle('Fixed transactions')/items
  ?$filter=WeekNumber eq 11 and Status ne 'CANCELLED'
  &$select=ClientID,TransactionType,Amount,AgentName,Status
  &$orderby=SubmitDate desc

Authentication uses the user's existing SharePoint session token — no additional OAuth configuration required for users already logged into Microsoft 365. External sharing is not enabled; all access is behind Microsoft Entra ID (formerly Azure AD).

How agent data isolation works

SharePoint's audience targeting combined with a filtered list view means that when an agent opens the Agent Dashboard, the REST API query automatically appends their Microsoft 365 user identity to the filter: Agent1Name eq '[current user]' OR Agent2Name eq '[current user]' and so on through all split positions. An agent with a 25% split on a transaction sees that transaction in their dashboard. An agent with no credit on a transaction never sees it.

Sensitive data handling

SSN and DOB fields (stored in the Clients list for BD/IA suitability compliance) are restricted to Operations Manager and Administrator roles only. These fields are excluded from all agent-facing views and from the executive dashboard. SharePoint column-level permissions enforce this at the API layer — not just in the UI.
